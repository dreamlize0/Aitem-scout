# AI템 스카우트 (AItem Scout) 프론트엔드-백엔드 협업 개발 계획서

본 문서는 'AItem Scout' 프로젝트의 원활한 병렬 개발을 위해 작성된 프론트엔드(Antigravity)와 백엔드(Claude) 간의 협업 표준 지침입니다. 개발 간 충돌을 방지하고 책임 소재를 명확히 하여 개발 속도와 안정성을 높이는 것을 목적으로 합니다.

## 1. 역할 분담 및 기술 스택

### 1.1 프론트엔드 (Antigravity 담당)
- **핵심 역할:** UI/UX 구현, 상태 관리, 백엔드 API/Supabase 연동, 사용자 인터랙션 처리
- **기술 스택:** React (Next.js 권장), Vanilla CSS (또는 Tailwind CSS), Supabase Client
- **주요 작업 내용:**
  - 다크 테마 기반 반응형 UI 구성 (#121214 배경, Sky Blue 주조색, Neon Green 포인트 등 프리미엄 디자인)
  - 검색 대시보드, 다중 필터 UI, 로딩 스켈레톤 애니메이션('AI 스카우팅 중...') 구현
  - 검색 결과 및 AI 리포트 렌더링, 외부 출처 Out-link 아이콘 처리
  - 보관함/프로젝트 폴더 관리 UI 및 공유 링크 기반 뷰어 페이지 구현

### 1.2 백엔드 (Claude 담당)
- **핵심 역할:** 비즈니스 로직, 데이터 수집 파이프라인(API 연동), LLM 프롬프트 엔지니어링, DB 스키마 설계
- **기술 스택:** Supabase (PostgreSQL, Edge Functions), Node.js/Python, LLM (Claude/GPT-4o), 외부 데이터 API
- **주요 작업 내용:**
  - Supabase DB 스키마 설계 및 보안(RLS) 정책 설정
  - 외부 검색 API(국내/해외/SNS) 모듈형 연동 및 데이터 수집 로직 구현
  - LLM 기반 데이터 분석, 타겟 맞춤형 추천 사유 및 리포트 생성 파이프라인 구축
  - API Rate Limit 대응을 위한 검색 결과 캐싱 처리 (12~24시간 TTL DB 저장)

## 2. 데이터베이스 스키마 및 API 통신 규약

### 2.1 데이터/API 사전 합의 (Contract First)
백엔드 로직이 완성되기 전, 프론트엔드가 Mock 데이터로 작업할 수 있도록 API Request/Response JSON 스키마를 최우선으로 확정해야 합니다.
- `users`: 계정 및 워크스페이스 정보
- `projects`: 프로젝트(보관함) 정보 (공유용 난수화 토큰 포함)
- `search_logs` / `caches`: 외부 API 캐싱 데이터
- `saved_items`: 추천 아이템 상세 내용 (출처 URL, 메타데이터 포함)

### 2.2 API 응답 구조 표준 (JSON)
프론트엔드와 백엔드의 모든 통신은 아래 구조를 따르도록 통일합니다.
```json
{
  "status": "success" | "error",
  "data": { ... }, // 성공 시 반환되는 페이로드 (리스트, 객체 등)
  "error": { // 에러 시 반환되는 내용
    "code": "ERROR_CODE",
    "message": "사용자에게 표시될 수 있는 친절한 에러 메시지"
  }
}
```

## 3. 병렬 개발을 위한 페이즈별 진행 플랜

| 페이즈 | 백엔드 (Claude) 진행 항목 | 프론트엔드 (Antigravity) 진행 항목 | 동기화 포인트 |
| :--- | :--- | :--- | :--- |
| **Phase 1: 기틀 마련** | Supabase 인프라 세팅, DB 스키마 설계(SQL), API 스펙 명세서 작성 | 프론트엔드 프로젝트 셋업, 라우팅 및 기본 레이아웃 구성, 공통 UI 컴포넌트 개발 | DB 스키마 및 API Request/Response JSON 구조 합의 |
| **Phase 2: 검색 엔진** | 외부 API 연동, 검색 결과 캐싱, LLM 프롬프트 초안 작성 및 테스트 | 대시보드 및 필터 UI, API 상태 관리(Zustand 등), 3~5초 대기 스켈레톤 애니메이션 적용 | 장기 대기(Latency)에 따른 프론트-백엔드 타임아웃/에러 처리 테스트 |
| **Phase 3: 보관 및 공유** | LLM 리포트 출처 매핑 로직, 보관함 저장 API, 고유 URL 생성 및 비로그인 조회 API 구현 | 검색 결과 카드/리포트 뷰어 구현, 폴더 UI, 공유 웹뷰 페이지 개발 | 출처 아이콘/링크 렌더링 검증, 비로그인 웹뷰 권한(RLS) 테스트 |
| **Phase 4: 폴리싱** | 외부 API 차단 및 Rate Limit 발생 시 대체/예외 처리 로직(부분 성공) 강화 | 다크테마 시각적 완성도(마이크로 애니메이션 등) 향상, 반응형 QA | 통합 테스트 및 네트워크 지연, API 에러 상황 모의 훈련 |

## 4. 충돌 방지 및 협업 가이드라인

1. **API Mocking 최우선 개발:** 프론트엔드는 백엔드 로직이 배포되는 것을 기다리지 않고, 정의된 JSON 스키마를 기반으로 Mock API를 연동하여 뷰와 상태 로직을 완성합니다.
2. **독립된 폴더 구조 사용:** 
   - 프론트엔드 코드는 루트 디렉토리 또는 `/frontend`
   - 백엔드 코드(Supabase 엣지 펑션, DB 마이그레이션 등)는 `/supabase` 로 엄격히 분리하여 Git 충돌을 막습니다.
3. **Graceful Degradation (부분 성공 구조):**
   - 인스타그램이나 네이버 등 특정 API 연동에 실패하더라도 백엔드 서버 전체가 다운되지 않고, 성공한 데이터만 반환합니다.
   - 프론트엔드는 일부 데이터가 누락되더라도 화면이 깨지지 않고 유연하게 표시되도록 방어 코드를 작성합니다.
